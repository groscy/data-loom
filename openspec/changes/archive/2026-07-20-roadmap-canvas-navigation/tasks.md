## 4. Pan, zoom, and controls (public/app.js)

- [x] 4.1 Drag-to-pan: `pointerdown` on `.board-wrap` (ignore presses that originate on a `.gcard` or a control), `pointermove` to translate, `pointerup`/`pointercancel` to release. Use pointer capture, and set `will-change: transform` only for the duration of the drag.
- [x] 4.2 Wheel: bare wheel/trackpad pans both axes; `ctrlKey`/`metaKey` + wheel zooms. Register with `{ passive: false }` and `preventDefault()` so the page does not scroll. — zoom factor is `exp(-dy * 0.0015)` rather than a fixed step, so a trackpad pinch (which arrives as ctrl+wheel) feels continuous; `deltaMode === 1` is normalised from lines to px.
- [x] 4.3 Pointer-anchored zoom helper `zoomAt(clientX, clientY, factor)`: convert the pointer to canvas coordinates, clamp the new `k` to `[MIN_K, MAX_K]`, translate so that canvas point stays under the pointer. — the floor is `minScale() = min(MIN_K, fitScale())`, not a bare `MIN_K`: on a plan too wide to fit at 0.25 a hard floor would make "zoom out" clamp *above* the current scale and act like zoom in.
- [x] 4.4 `fitView()`: `k = min(vw / canvasW, vh / canvasH, 1)` clamped to range, centered in the viewport. The `min(…, 1)` keeps a small plan at 1:1 and centered rather than magnified. — returns `false` when the viewport is 0×0 (roadmap tab hidden) so a pending fit is not wrongly marked done.
- [x] 4.5 Wire the zoom-in / zoom-out / fit controls to `zoomAt` (anchored at the viewport center) and `fitView`.
- [x] 4.6 Keyboard on the focused `.board-wrap`: arrows pan by a fraction of the viewport, `+`/`=` and `-` zoom by a fixed ratio anchored at center, `0` fits. `preventDefault()` so arrows do not scroll an ancestor.
- [x] 4.7 Call `fitView()` on the first render of a project's roadmap (`!viewFitted`) and mark it fitted.
- [x] 4.8 Add a `ResizeObserver` on `.board-wrap` that re-clamps the pan, re-evaluates minimap visibility, and updates the viewport rectangle on window resize. — also carries a pending initial fit: a model that arrives while the roadmap tab is hidden fits on the first real size the observer reports.

## 8. Fix connector anchoring (pre-existing, found during 7.7)

## 9. Minimap visibility rule (amended after review)

- [x] 9.1 Overflow-only visibility made the minimap effectively unreachable: the canvas is `300 + (phases − 1) × 260` wide and never shorter than 600px, so a maximized 1920×1080 window needs **7 phases** to overflow horizontally, or a window under ~706px tall to overflow vertically. Real plans here run 1–4 phases, so the control existed but was never seen. Rule is now `phaseCount > 1 || canvasW > vw || canvasH > vh`, hiding only for a single-phase plan that fits.
- [x] 9.2 Track `phaseCount` in module scope alongside `canvasW`/`canvasH`; set it in `renderBoard` and zero it in `resetCanvasToViewport`.
- [x] 9.3 Amend the `Minimap overview of the whole roadmap` requirement and its scenarios — the previous "hidden when everything fits" scenario asserted the opposite of the new behaviour. Split into *shown for a multi-phase roadmap*, *appears for an oversized canvas*, and *hidden for a single-phase roadmap that fits*. Record the reasoning in `design.md`.
- [x] 9.4 Verify. — `sound_alchemist` (2 phases, 560×600 canvas fitting a 1356×794 viewport): minimap **shown**, 121×130 box, 4 `.mm-card` + 2 `.mm-frame`. `agora` (2 phases): shown. `cortex_stack` (1 phase, fits): **hidden**. Still fully functional on a fitting canvas — at fit the viewport rect correctly exceeds the canvas (everything visible), shrinks to 678×397 at k=2, and clicking at 10% vs 90% width moves the view (x=566 vs x=−330). Fit restores the exact prior transform.

**Not verified in this environment** (this preview pane could not produce screenshots, `ResizeObserver` callbacks, or completed CSS transitions): live window-resize behavior (7.11) and the visual appearance of the controls and minimap. Worth one pass in a real browser before release.
